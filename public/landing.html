Architecture

Identity: users, sessions, login, signup
Authorization: `/o/authorize` + consent + one-time code
Token Service: `/o/token` with client validation + JWT issuance
Claims API: `/o/userinfo` from bearer access token
Provider Metadata: discovery + JWKS
Client Management: registration, admin approve/reject

Auth Flow

1. Client redirects browser to /o/authorize with client_id, redirect_uri, scope, state, nonce.
2. Provider validates client + redirect + scope, checks session, then moves to consent.
3. User clicks allow or deny on consent page.
4. On allow, provider stores one-time code and redirects to client callback with code and state.
5. Client backend calls /o/token with code + client credentials.
6. Provider validates grant and returns access_token + id_token.
7. Client can call /o/userinfo using bearer access token.
